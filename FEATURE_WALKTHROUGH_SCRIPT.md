# Feature Walkthrough Script

Use this script during live technical interviews or product demos to highlight the enterprise-grade complexity of the system.

## Scene 1: The Adaptive AI Engine
**Action:** Log in as a Student and navigate to the Dashboard.
**Script:**
> "Welcome to the Student Dashboard. Notice how the 'Neural Gap Priority' widget isn't just static data—it is dynamically generated by our Spring Boot Adaptive Engine. The system continuously analyzes every question the student answers, calculates an average response time, and groups failures by Topic. When I click 'Start Adaptive Quiz', the backend algorithm actively filters the MySQL database to assemble a personalized payload targeting exactly the concepts this student is failing at."

## Scene 2: The Real-Time Leaderboard (WebSocket Federation)
**Action:** Start a live quiz session. Have two browser windows open side-by-side representing two different students.
**Script:**
> "Here we have a live multiplayer quiz session. As Student A answers a question, watch Student B's screen. The leaderboard updates instantaneously. What's impressive here isn't just that it uses WebSockets—it's *how* it uses them. We've bypassed Spring's simple in-memory broker and integrated a full STOMP Broker Relay via RabbitMQ. This means we can scale out to 100 backend servers behind an API Gateway, and this real-time sync will never break."

## Scene 3: The JWT Refresh Flow (Security Resilience)
**Action:** Open the Chrome DevTools Console. Manually alter the `jwt_token` in localStorage to simulate an expiration. Click a button that triggers an API call.
**Script:**
> "Security and UX rarely go hand-in-hand, but here we have a seamless Refresh Token flow. I've just forced my Access Token to expire. When I click this button, the request fails with a 401. However, our Axios interceptor catches the 401, pauses the execution queue, hits our backend `/refresh` endpoint, swaps the long-lived refresh token for a new access token, and replays the original request. The user experiences absolutely zero interruption, and they are never randomly logged out."

## Scene 4: Failure Handling & Skeletons
**Action:** Throttle the network to "Slow 3G" in DevTools, or simply kill the backend server.
**Script:**
> "An enterprise system shouldn't crash just because a service is down. Notice our System Status Indicator in the bottom corner—it actively pings the `/actuator/health` endpoint. If the server drops, it flashes red. When we navigate to the Analytics page under a slow network constraint, we don't get a blank screen. Our Framer Motion Skeleton Loaders mount immediately, providing a perceived performance boost while the backend computes the heavy SQL aggregations."
